# 텔레그램 입고 자동화 시스템

텔레그램 그룹방에 들어오는 외부업체의 파일 입고 알림을 Google Sheets 편성표와 연결해, 프로그램별 파일이 모두 들어왔는지 자동으로 판단하는 자동화 프로젝트입니다.

## 데모 사이트

이 저장소는 GitHub Pages로 바로 배포할 수 있는 발표용 웹 슬라이드를 포함합니다.

- 사이트 루트: `docs/`
- 메인 파일: `docs/index.html`
- 원본 PDF: `docs/assets/source.pdf`
- 슬라이드 이미지: `docs/assets/slides/`

로컬에서 확인:

```powershell
cd docs
python -m http.server 4173 --bind 127.0.0.1
```

브라우저에서 `http://127.0.0.1:4173/`을 열면 됩니다.

## 해결하려는 문제

현재 업무에서는 텔레그램 입고 알림을 사람이 보고 엑셀 또는 Google Sheets 편성표에 수동으로 반영합니다. 하지만 실제 운영에서는 다음 문제가 자주 생깁니다.

- 빠르게 여러 메시지가 들어오면 누락되거나 중복 기록될 수 있음
- 텔레그램 경로의 프로그램명과 편성표 프로그램명이 다를 수 있음
- 한 프로그램이 여러 회차 파일로 들어올 수 있음
- `23-24`처럼 회차 범위가 있으면 파일이 모두 들어왔는지 다시 세야 함
- 완료 시각은 첫 파일 시간이 아니라 마지막 파일 발생시간이어야 함

핵심은 파일을 받았는지가 아니라, **프로그램별로 받을 파일이 전부 들어왔는지 정확히 판단하는 것**입니다.

## 전체 구조

```text
Telegram
  -> Cloudflare Worker
  -> Google Apps Script Webhook
  -> Google Sheets
```

- **Telegram**: 외부업체가 입고 완료 메시지를 보내는 채널
- **Cloudflare Worker**: Telegram webhook과 Apps Script 사이의 프록시
- **Google Apps Script**: 메시지 파싱, 중복 방지, 편성표 비교, 상태 갱신
- **Google Sheets**: 운영자가 보는 편성표, 입고로그, 미매칭검수, 대시보드

## Google Sheet

운영 시트:

https://docs.google.com/spreadsheets/d/1ZNPijMHvyZXXKVdr72q4DEEi8YJxpY0RU_IT_Y0xS14/edit

주요 시트 역할:

- **편성표**: 채널, 프로그램, 회차, 파일 입고시간 등 최종 운영 상태를 보는 시트
- **입고로그**: 텔레그램에서 수신한 모든 메시지와 처리 결과를 저장하는 시트
- **미매칭검수**: 자동으로 확정하기 어려운 메시지를 모아 운영자가 직접 편성표 항목을 선택하는 시트
- **대시보드**: 전체 프로그램 수, 완료, 입고중, 미입고, 확인필요 상태를 요약하는 시트
- **테스트메시지**: 텔레그램에 복사해 테스트할 수 있는 샘플 메시지와 예상 결과

자세한 시트 구성은 [spreadsheet/README.md](spreadsheet/README.md)를 참고하세요.

## 메시지 형식

기본적으로 아래와 같은 텔레그램 메시지를 처리합니다.

```text
[완료]
파일명:24.mpg
경로:tvN/시리즈/영상/어서와~ 한국은 처음이지?/24.mpg
발생시간: 2026-05-10 09:02:00
```

## 자동 처리 알고리즘

1. 텔레그램 메시지에서 파일명, 경로, 발생시간, 메시지 ID를 추출합니다.
2. 같은 채팅방 ID와 메시지 ID는 한 번만 처리합니다.
3. 같은 프로그램키와 파일명이 이미 입고로그에 있으면 중복 파일로 보고 다시 쌓지 않습니다.
4. 경로 전체에서 편성표에 있는 프로그램명을 찾습니다.
5. 프로그램명이 정확히 매칭되면 편성표의 회차를 기준으로 필요한 파일 수를 계산합니다.
6. 서로 다른 파일명이 예상 파일 수만큼 들어오면 상태를 완료로 바꿉니다.
7. 파일 입고시간에는 해당 프로그램에서 가장 늦은 발생시간을 `HH:mm` 형식으로 기록합니다.
8. 자동 확정이 어려운 메시지는 미매칭검수 시트로 보냅니다.

## 회차별 파일 수 판단

편성표의 회차 값으로 예상 파일 수를 계산합니다.

- `23` -> 파일 1개 필요
- `23-24` -> 23, 24회차이므로 파일 2개 필요
- `181~` -> 끝 회차가 아직 없으므로 완료 판단에서 제외하거나 확인필요로 관리

완료 조건은 단순히 메시지가 왔는지가 아니라, **서로 다른 파일명이 필요한 개수만큼 도착했는지**입니다.

## 예외 처리

자동화가 확정할 수 없는 메시지는 실패로 버리지 않고 검수 대상으로 보관합니다.

- 프로그램명이 편성표와 다를 때
- 경로 구조가 예상과 다를 때
- 끝 회차가 아직 확정되지 않았을 때
- 같은 메시지 또는 같은 파일명이 반복 도착했을 때
- 빠르게 여러 메시지가 연속으로 들어올 때

운영자는 미매칭검수 시트에서 텔레그램 명칭을 보고 편성표 항목을 선택할 수 있습니다. 선택된 항목은 현재 파일 카운트에도 즉시 반영되도록 설계되어 있습니다.

## 포함 파일

```text
apps-script/Code.gs       # Google Apps Script Webhook 코드
docs/                     # GitHub Pages 정적 사이트
public/pdf-slides/        # 로컬 웹 슬라이드 원본 작업물
spreadsheet/README.md     # Google Sheets 구성 설명
```

## GitHub Pages 배포 방법

1. GitHub 저장소를 생성합니다.
2. 이 프로젝트를 push합니다.
3. 저장소의 **Settings -> Pages**로 이동합니다.
4. Source를 `Deploy from a branch`로 설정합니다.
5. Branch는 `main`, Folder는 `/docs`를 선택합니다.
6. 저장하면 `https://<계정명>.github.io/<저장소명>/` 주소로 배포됩니다.

## 보안 주의

봇 토큰은 저장소에 올리면 안 됩니다. 이미 노출됐을 가능성이 있다면 Telegram BotFather에서 `/revoke`로 기존 토큰을 폐기하고 새 토큰을 발급받는 것을 권장합니다.
